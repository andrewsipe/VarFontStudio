Same underlying data (STAT / fvar / name tables, two font revisions), three different information architectures. Switch between them above. All three keep values in monospace, color-code changes consistently (amber = modified, green = added, red = removed), and default to hiding untouched rows so the eye lands on what actually changed.

Why these three, and what each trades off

  • 1 · Grouped columns keeps your original before/after column metaphor but adds collapsible groups, a sticky "diffs only" filter, and a left-edge color gutter instead of full-row tinting. Closest to what you have, least risk, still keeps the file structure (STAT/fvar/name) visible at all times.
  • 2 · Sidebar master-detail moves table/section navigation into a sidebar with change counts, so you're never scrolling past thousands of unchanged rows to reach the next diff — you jump straight to "name" or "fvar instances." Best if files get large (many instances, many locales in name table).
  • 3 · Unified diff list drops the two-column framing entirely and shows one line per changed field as old → new, git-diff style, grouped by table. Fastest to scan for "what actually changed" across an entire comparison, but loses the side-by-side full-table view — you'd want a "show all fields" toggle to fall back to columns when needed.
  • All three assume unchanged rows are collapsed/hidden by default — that was the biggest legibility problem in the screenshot (walls of identical text competing visually with the handful of real diffs).
